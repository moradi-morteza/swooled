بررسی افزایش و کاهش حافظه در Swoole Redis Server

گاهی بعد از ایجاد تعداد زیادی Connection آزمایشی، مقدار RAM مصرفی برنامه افزایش پیدا می‌کند. بعد از بسته شدن Connectionها، حافظه کاهش پیدا می‌کند اما دقیقا به مقدار اولیه برنمی‌گردد. این رفتار در بسیاری از برنامه‌های نوشته شده با Swoole و PHP طبیعی است.

چرا حافظه به مقدار اول برنمی‌گردد؟

  • PHP Memory Allocator: حافظه آزاد شده را معمولا برای استفاده‌های بعدی نگه می‌دارد.
  • Swoole Memory Pool: برای افزایش سرعت، بعضی منابع را cache می‌کند.
  • Kernel Socket Buffer: اطلاعات مربوط به TCP Connection ممکن است کمی دیر آزاد شوند.
  • Memory Fragmentation: باعث می‌شود فضای آزاد وجود داشته باشد ولی به سیستم عامل برنگردد.

رفتار طبیعی چگونه است؟

مثال:
30MB → 120MB → 45MB

اگر چند بار تست کنیم و همیشه حدود 45MB باقی بماند، این معمولا نشانه Leak نیست؛ بلکه حافظه برای استفاده مجدد نگه داشته شده است.

رفتار خطرناک چیست؟

مثال:
30MB → 120MB → 80MB
تست بعدی:
80MB → 160MB → 120MB

اگر بعد از هر تست مقدار پایه بیشتر شود، احتمال Memory Leak وجود دارد و باید بررسی شود.

چگونه در سیستم خودمان شبیه‌سازی کنیم؟

با ایجاد تعداد زیادی Connection و نگه داشتن آن‌ها برای چند ثانیه:

open connections → hold → close → repeat

نمونه تست:

500 connection هر ثانیه نگهداری 5 ثانیه مدت تست: 60 ثانیه

چگونه مقدار حافظه را بررسی کنیم؟

docker stats ps aux | grep server.php

در محیط Kubernetes معمولا Grafana مقدار RSS یا Container Memory را نشان می‌دهد، نه فقط آبجکت‌های زنده PHP.

آیا می‌توانیم حافظه را دقیقا به مقدار اول برگردانیم؟

معمولا خیر. تنها راه قطعی Restart کردن Process یا Container است.

راهکارهای کنترل حافظه

  • تنظیم max_request برای recycle شدن Workerها
  • استفاده از Memory Limit در Docker/Kubernetes
  • Restart کردن Worker در صورت عبور از Threshold
  • مانیتور کردن روند حافظه، نه فقط یک مقدار لحظه‌ای

نتیجه‌گیری

افزایش RAM بعد از فشار تست و برگشت به یک مقدار کمی بالاتر، در Swoole معمولا رفتار طبیعی است. مهم این است که حافظه در هر تست به صورت دائمی افزایش پیدا نکند.
